# GD32驱动备忘录

## 20250419

调整中断优先级，对于轮询，且不包含原语的事件，优先级打到最低。

对于中断密度高的通信控制器，将优先级打在第二档。
对于中断密度低的通信控制器，将优先级打在第三档。

优先级最高的事件，为唤醒事件，必须能够打断从任何中断进入的睡眠。 亦或是进入睡眠的函数只能在低优先级的中断中或是主函数中被调用。

## 20250430

- [ ] 编写程序时，需要对所使用变量的使用环境属性，变量声明位置属性，在变量命名时一并进行说明。
    - 例如：在临时变量的命令名中添加temp标识，表明该变量的生存周期只在该函数内。
    - 函数的传入参数命名，需要在变量名开头添加下划线'_'
    - 全局变量的命名，必须包含g_开头

- [ ] 功能块的执行结果需要遵循一定范式进行传递, 若在同一份工程中存在多种传递方式，可能会导致人为的编程失误，造成一定的调试困难。使得功能块的执行结果没有被正确响应，相当于功能块失效。

- [ ] 当某个功能块失效时，可以围绕以下几点进行分析
    - 功能块逻辑检查
    - 功能块的传入参数是否被正确使用
    - 功能块的执行结果是否正确传出。
        > 近期遇到的异常情况：Modbus tcp某个功能块的执行结果被赋值到局部变量中，导致该功能块失效

## 20250529

- [ ] 目前的睡眠设计方案有概率出现无法唤醒的情况，怀疑是芯片内部的中断响应调度问题。唤醒中断，应该放在最高优先级，其次才是其他的中断。
- [ ] 正交编码器和i2c控制器之间的干扰依旧未得到确认。目前可确认的信息：
        正交编码器使用中断处理，当i2c通信频次低时，编码器的采集正常工作；
        当i2c通信频次高时，编码器的采集无法正常工作，中断会重复进入数次。
        从硬件的角度上来看，编码器AB相波形非常干净，并没有受到i2c的串扰。
